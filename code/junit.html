Repository
A repository is managing the persistence of entities.

Repository naming ---WorkItemRepository
ancillary unit
Ancillary units are put to the same package as the unit they support.
If the design or needs of testing mandates it, a ---DTO may have a Factory
---an Entity must have a Repository
all DTO, Entity, Service and Controller units must have a Test Data
all Service and Controller units must have a Test Base
all Service and Controller units must have a Stub Provider
---a legacy object may have a Factory and/or a Stub Provider and/or a Test Data. Those should be put in a package named 'legacy' under the main package of the project.

repository
  • The java implementation of a repository is an interface extending PagingAndSortingRepository<EntityType, Long>, where EntityType is the name of the Entity unit it handles.
  • A method name of the Repository can start with find. It returns List<EntityType>
  • A method name of the repository can start with get. It returns Optional<EntityType>
  • A method name of the repository can start with delete. It returns void
  • A method name of the repository can start with count. It returns long
  • A method name of the repository can start with exist. It returns boolean
  • The repository have the methods defined as Queryes contained by the repository. All the method names directly define a query as described in https://docs.spring.io/spring-data/jpa/docs/current/reference/html/#repositories.query-methods.details
  • The default repository methods can be used as prescribed in the model

Delegate

Delegates are to implement classes containing both state and business logic, and/or when the DI framework needs to be manually bootstrapped.

The state is delegated to a DTO

The business logic is delegated to Services

Rules:

generated

  • Where all the information necessary to write a unit exists in the model, the unit will be generated in the long term
  • If such a unit is missing, it is the responsibility of the coder to create it.

interface class

  • Units without naming convention (Delegate, Configuration provider, and units which part of the boilerplate needed with the frameworks used) should have an @InterfaceClass annotation, denoting the lack of naming convention applicable.

modeled placement

First-class production code units are placed in the production code hierarchy under the package designated by the design

production code

production units are placed in the production code hierarchy

basic naming conventions

  • modules are files named with CamelCase, with the language-specific extension (e.g. RegistrationController.java or RegistrationController.py)
  • non-module units (classes, interfaces and enums) are named with CamelCase. There is only one unit in a module.
  • methods are named with lowerCaseCamelCase
  • parameters, fields and local variables are named with lowerCaseCamelCase
  • constants are named with SNEAK_CASE
  • references to code units must be named after the name of the code unit with the first letter in lowercase, eg registrationController

class

The java implementation of the unit is a class

Delegate rules

In Java:

  • The DI bootstrapping is delegated to a static void method with sole parameter of this in an interface of the underlying service.
  • The state of a delegate is stored in a DTO, referenced in a field annotated with @Delegate and if the integration tests need it, @Getter annotation.
  • The underlying services are referenced using @Autowire -d fields using the interface
  • The methods use the delegate pattern, with this as an extra parameter in the delegated call.
